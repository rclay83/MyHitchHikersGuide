Adding A Feature

  • Programming By Difference

    Problem: Need to add new behavior to an existing method. New behavior is alternative to existing behavior.

    Algorithm:

    1. Subclass the class in which you want to make a change in a method.
    2. Make sure the super class method visibility is at least protected.
    3. Override the method in the subclass and use TDD to develop it with the new behavior.
    4. Once the test is well defined and passing on the subclass, comment out the override in the subclass and refactor the original method to include logic for the new case. The existing test will call the old method with the new logic now. (1)
    5. Remove no longer needed subclass.

    Beware of the Liskov Substitution Principle when subclassing

    Notes:

    1. An alternative to adding new logic to the old method is to encapsulate the different behavior in an interface. Use the strategy pattern to encapsulate the differing algorithms and call an interface method to carry out the behavior.

  • Liskov Substitution Principle

    Objects of subclasses should be substitutable for objects of their superclasses throughout the code base. If they aren't there could be silent errors present.

    1. Whenever possible, avoid overriding concrete methods. If inheritance is desired, make the class containing the method to be overriden an abstract class and the method abstract. Implement explicitly in a subclasses.
    2. If you do, see if you can call the method you are overriding in the overriding method.

    Example: Rectangle class with setHeight, setWidth, and getArea methods. Square, a subclass of Rectangle, in which getArea is called with different values for height and width. What value should be returned for the area?


    Problem: concrete methods that are overriden can cause confusing behavior in clients that aren't aware the method is overriden.
